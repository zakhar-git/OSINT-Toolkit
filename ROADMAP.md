# Roadmap

[Previous: Changelog](CHANGELOG.md) · [Next: README](README.md) · [Back to README](README.md)

The roadmap prioritizes verification depth and maintainability over catalog size. Dates are intentionally omitted until maintainers commit to a release window.

## Now

- Establish the first public review cycle for all tool cards.
- Create a quarterly maintenance audit process for commits, releases, issues, documentation, and service availability.
- Configure private vulnerability reporting and repository contact ownership.
- Test installation examples across Linux, macOS, and Windows where supported.
- Add stable release tags and signed changelog entries.

## Next

- Add machine-readable catalog data with a documented schema.
- Add per-entry verification evidence and upstream release monitoring.
- Add a rejected-tools register with reasons and replacement candidates.
- Add workflows for video verification, public records, and web archiving.
- Add translations only after the English information architecture stabilizes.

## Later

- Publish a static documentation site from the same Markdown sources.
- Add optional offline bundles for field environments.
- Add a transparent deprecation dashboard and scheduled re-verification queue.
- Define reproducible test fixtures for CLI examples.

## Not planned

- Ranking tools by popularity alone.
- Hosting breach data, credentials, private datasets, or target dossiers.
- Publishing bypass, evasion, exploitation, or unauthorized-access playbooks.
- Automatically importing unreviewed entries from other lists.

## Decision criteria

A roadmap item advances when it improves source quality, analyst safety, reproducibility, navigation, or maintenance cost without expanding harmful collection capability.
